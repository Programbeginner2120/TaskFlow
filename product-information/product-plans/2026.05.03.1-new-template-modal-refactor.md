# Plan: New Template Modal — Edit Mode Refactor

## Overview

Allow users to open an existing template for editing by clicking it in the sidebar. The same `NewTemplateModalComponent` is reused in edit mode: it pre-populates with the template's current data and calls `updateTemplate()` instead of `addTemplate()` on submit.

The approach uses Angular's component lifecycle as the reset mechanism (**Option C3**): the parent wraps the modal in `@if (addingTemplate())` so the component is destroyed on close and born fresh on open. This eliminates the need for any `effect`, `ngOnChanges`, or manual reset logic inside the modal.

No new components or files are created. The changes are confined to:

1. `rrule.util.ts` — add `parseRule()` utility
2. `new-template-modal.component.ts` — remove `isOpen` input; add optional `template` input; populate signals in `ngOnInit`; branch submit to create vs. update
3. `new-template-modal.component.html` — pass `[isOpen]="true"` to inner `ModalComponent`; conditional title and button label; updated click binding
4. `landing-page.component.ts` — add `selectedTemplate` signal and `editTemplate()` method
5. `landing-page.component.html` — wrap modal in `@if`; wire click on template items

---

## Background: Current State

### `NewTemplateModalComponent`

- Inputs: `isOpen = input.required<boolean>()`, `close = output<void>()`
- All form state is initialised to defaults in signal declarations and never reset
- `createTemplate()` always calls `taskListTemplateStateService.addTemplate(request)` then emits `close`
- No concept of an "existing" template

### `LandingPageComponent`

- `addingTemplate = signal<boolean>(false)` controls whether the modal is open
- `startAddingTemplate()` / `stopAddingTemplate()` set that signal
- Template items in the sidebar have no `(click)` handler

### `ModalComponent` (inner shared component)

- Uses `@if (isOpen())` internally to conditionally render the backdrop and dialog
- Has no CSS enter/leave transitions — content appears and disappears immediately
- Body scroll lock is driven by the `isOpen` input via an `effect`

### Why C3 works here

Because `ModalComponent` already destroys its DOM with `@if (isOpen())`, there is no animation to protect. Moving the `@if` one level up to the parent — on `<app-new-template-modal>` itself — gives the same visual result while letting Angular's component lifecycle handle state initialisation for free. The inner `ModalComponent` always receives `[isOpen]="true"` since the outer `@if` already controls existence.

### `TaskListTemplateStateService`

- `updateTemplate(id, body)` calls the PUT endpoint and patches the `_templates` signal in the `tap`
- Returns `void` (fire-and-forget with `subscribe({ error: console.error })`)

## Rrule Parsing Requirement

When the modal opens for an existing template it must reverse-engineer the stored rrule string back into the three UI selectors (frequency, day-of-week, day-of-month). The rrule shapes that the app currently produces are:

| Rrule string | Frequency | Day |
|---|---|---|
| `FREQ=DAILY` | `DAILY` | — |
| `FREQ=WEEKLY;BYDAY=MO` | `WEEKLY` | `MO` |
| `FREQ=MONTHLY;BYMONTHDAY=15` | `MONTHLY` | — (monthDay = 15) |

A small `parseRule(rrule: string)` utility function will be added to `rrule.util.ts` that returns `{ frequency, dayOfWeek?, monthDay? }`.

## Step 1 — Add `parseRule` to `rrule.util.ts`

**File:** `src/app/utils/rrule.util.ts`

Add a function below the existing `buildRule` that parses the three possible rrule shapes back into UI-friendly values:

```typescript
export function parseRule(rrule: string): {
    frequency: RruleFrequency;
    dayOfWeek?: RruleDay;
    monthDay?: number;
} {
    const freq = /FREQ=(\w+)/.exec(rrule)?.[1] as RruleFrequency ?? RruleFrequency.WEEKLY;
    const byday = /BYDAY=(\w+)/.exec(rrule)?.[1] as RruleDay | undefined;
    const bymonthday = /BYMONTHDAY=(\d+)/.exec(rrule)?.[1];
    return {
        frequency: freq,
        dayOfWeek: byday,
        monthDay: bymonthday ? Number(bymonthday) : undefined,
    };
}
```

- Add `OnInit` to the `@angular/core` import
- Add `parseRule` to the `rrule.util` import

### 2.2 Replace `isOpen` input with optional `template` input

Remove:
```typescript

```

Add:
```typescript
readonly template = input<TaskListTemplate | null>(null);
```

When `null` (default), the modal is in create mode. When set, the modal is in edit mode.

### 2.3 Add a computed `isEditMode`

```typescript
readonly isEditMode = computed<boolean>(() => this.template() !== null);
```

### 2.4 Implement `ngOnInit` to populate signals in edit mode

The component is always created fresh (via `@if` in the parent), so `ngOnInit` runs exactly once per open. Signal field declarations retain the existing create-mode defaults; `ngOnInit` overrides them only when a template is provided.

Add `implements OnInit` to the class declaration and add:

```typescript
ngOnInit(): void {
    const t = this.template();
    if (!t) return; // create mode — defaults already set in signal declarations

    this.templateName.set(t.name);
    this.templateColor.set(t.color as TaskListTemplateColor);
    this.templateTasks.set(
        t.taskTemplates.map(tt => ({
            title: tt.title,
            notes: tt.notes ?? '',
            dueDateOffset: tt.dueDateOffset,
            subtaskTemplates: tt.subtaskTemplates.map(st => ({ title: st.title })),
        }))
    );
    const parsed = parseRule(t.rrule);
    this.templateFrequency.set(parsed.frequency);
    if (parsed.dayOfWeek) this.dayOfWeek.set(parsed.dayOfWeek);
    if (parsed.monthDay)  this.monthDay.set(parsed.monthDay);
}
```

No `constructor`, no `effect`.

### 2.5 Replace `createTemplate()` with `submitTemplate()`

Rename the existing `createTemplate()` method to `submitTemplate()`. After building the rrule and request body, branch on `isEditMode()`:

```typescript
submitTemplate(): void {
    this.submitted.set(true);
    if (!this.isFormValid()) return;

    // ... existing rrule building logic unchanged ...

    if (this.isEditMode()) {
        const updateRequest: UpdateTaskListTemplateRequest = {
            name: this.templateName(),
            color: this.templateColor(),
            rrule,
            timezone: Intl.DateTimeFormat().resolvedOptions().timeZone,
            taskTemplates: this.templateTasks(),
        };
        this.taskListTemplateStateService.updateTemplate(this.template()!.id, updateRequest);
    } else {
        const createRequest: CreateTaskListTemplateRequest = {
            name: this.templateName(),
            color: this.templateColor(),
            rrule,
            timezone: Intl.DateTimeFormat().resolvedOptions().timeZone,
            taskTemplates: this.templateTasks(),
        };
        this.taskListTemplateStateService.addTemplate(createRequest);
    }

    this.close.emit();
}
```

## Step 4 — Update `LandingPageComponent`

**File:** `src/app/pages/landing-page/landing-page.component.ts`

### 4.1 Add `selectedTemplate` signal

Add `TaskListTemplate` to the import from `../../interfaces/task.interface`, then add:

```typescript
selectedTemplate = signal<TaskListTemplate | null>(null);
```

### 4.2 Add `editTemplate()` method

```typescript
editTemplate(template: TaskListTemplate): void {
    this.selectedTemplate.set(template);
    this.addingTemplate.set(true);
}
```

### 4.3 Update `stopAddingTemplate()` to clear `selectedTemplate`

```typescript
stopAddingTemplate(): void {
    this.addingTemplate.set(false);
    this.selectedTemplate.set(null);
}
```

Clearing `selectedTemplate` here is a hygiene measure — because the component is destroyed by the `@if`, it has no practical effect on state, but it keeps the parent signal tidy.

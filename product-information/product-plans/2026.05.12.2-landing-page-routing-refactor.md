# Landing Page Routing Refactor — Persist View on Reload

**Date:** 2026-05-12  
**Goal:** Preserve the active `AppView` (and selected list ID) across page reloads by encoding view state in the URL as query parameters.

---

## Problem

`LandingPageComponent` tracks the current view and selected list entirely via in-memory signals:

```ts
currentView   = signal<AppView>('my-day');
selectedListId = signal<number | null>(null);
```

On every reload both reset to their defaults, so the user always lands on **My Day** regardless of where they were.

---

## Chosen Approach: Query Parameters

Encode view state in the URL:

| View | URL |
|---|---|
| My Day | `/landing-page?view=my-day` |
| Upcoming | `/landing-page?view=upcoming` |
| Calendar | `/landing-page?view=calendar` |
| List (id=5) | `/landing-page?view=list&listId=5` |

**Why not child routes?**  
`LandingPageComponent` is a view-switching shell, not a router. Query params are semantically correct for persisting selection state. Child routes would require extracting `selectedTask` and `searchQuery` into a shared service, refactoring the template to use `<router-outlet>`, and touching routes/sidebar/event flow — a much larger blast radius for the same outcome.

---

## Implementation Plan

### 1. `app.routes.ts` — No changes required

The existing `landing-page` route is sufficient. Angular's router preserves query params natively.

---

### 2. `landing-page.component.ts`

#### 2a. Inject `ActivatedRoute` and `Router`

```ts
private readonly route  = inject(ActivatedRoute);
private readonly router = inject(Router);
```

#### 2b. Derive `currentView` and `selectedListId` from query params

Replace the two plain writable signals with computed signals driven by `toSignal(route.queryParamMap)`:

```ts
private readonly queryParams = toSignal(this.route.queryParamMap);

currentView = computed<AppView>(() => {
    const view = this.queryParams()?.get('view');
    return (view === 'my-day' || view === 'upcoming' || view === 'calendar' || view === 'list')
        ? view
        : 'my-day';
});

selectedListId = computed<number | null>(() => {
    const raw = this.queryParams()?.get('listId');
    const id = raw ? parseInt(raw, 10) : NaN;
    return this.currentView() === 'list' && !isNaN(id) ? id : null;
});
```

#### 2c. Replace `signal.set(...)` view mutations with `router.navigate`

All places that currently call `this.currentView.set(...)` become a `navigate` call instead:

**Nav items (navItems computed):**
```ts
navItemRouteFn: () => this.navigateTo('my-day')
navItemRouteFn: () => this.navigateTo('upcoming')
navItemRouteFn: () => this.navigateTo('calendar')
```

**`selectListView` method:**
```ts
selectListView(listId: number): void {
    this.router.navigate([], {
        relativeTo: this.route,
        queryParams: { view: 'list', listId },
        queryParamsHandling: 'merge'
    });
    this.sidebarComponent().isExpanded.set(false);
}
```

**Private helper:**
```ts
private navigateTo(view: AppView): void {
    this.router.navigate([], {
        relativeTo: this.route,
        queryParams: { view, listId: null },
        queryParamsHandling: 'merge'
    });
}
```

Setting `listId: null` in `navigateTo` removes the `listId` param when switching away from a list view.

#### 2d. Add new imports

```ts
import { ActivatedRoute, Router } from '@angular/router';
import { toSignal } from '@angular/core/rxjs-interop';
import { RouterModule } from '@angular/router';
```

`RouterModule` must be added to the component `imports` array so the template can use router directives if needed (safe to include even if unused in the template today).

---

### 3. Template — No changes required

`currentView()` and `selectedListId()` remain callable signals from the template's perspective. The `@switch` block and list binding continue to work identically.

---

## Acceptance Criteria

- Navigating to **My Day**, **Upcoming**, or **Calendar** updates the URL to `?view=<view>`
- Clicking a list updates the URL to `?view=list&listId=<id>`
- Reloading the page on any of those URLs restores the correct view and (for lists) the correct list
- Navigating to `/landing-page` with no query params shows My Day by default
- All other features (task details panel, search, add list, templates) are unaffected

Implementation Plan — Task Calendar View

This plan covers the full implementation of a responsive, monthly (and optionally weekly) task calendar view built on FullCalendar for Angular (@fullcalendar/angular, @fullcalendar/daygrid, @fullcalendar/interaction). The calendar displays tasks by their due date, caps visible events per cell at two with a "+N others" overflow link, opens a task-details side panel on event click, and is fully responsive via the existing PlatformService.

The mockup reference is product-information/mockup-images/full-calendar-with-tasks-mockup.png.

Step 2 — Map Task[] to FullCalendar EventInput[]

FullCalendar works with an array of EventInput objects. We derive this array from the reactive task signal so the calendar re-renders automatically when tasks change.

Computed signal — calendarEvents

readonly calendarEvents = computed(() =>
    this.taskService.tasks()
        .filter(t => t.dueDate !== null)
        .filter(t => {
            const q = this.searchQuery().toLowerCase().trim();
            return !q || t.title.toLowerCase().includes(q);
        })
        .map(t => ({
            id: String(t.id),
            title: t.title,
            start: t.dueDate!,          // FullCalendar accepts Date objects
            allDay: true,
            classNames: t.completed ? ['fc-event--completed'] : [],
            extendedProps: { task: t }, // store full task for click handler
        }))
);

Wire the events into CalendarOptions

CalendarOptions is built as a computed signal so that responsive options (e.g. headerToolbar layout, dayMaxEvents) react to platform changes. The events property is set to () => this.calendarEvents() (a function) so FullCalendar calls it when it needs to re-render rather than holding a stale snapshot. 

readonly calendarOptions = computed<CalendarOptions>(() => ({
    plugins: [dayGridPlugin, interactionPlugin],
    initialView: 'dayGridMonth',
    headerToolbar: {
        left:   this.platformService.isPhone() ? 'prev,next' : 'prev,next today',
        center: 'title',
        right:  this.platformService.isPhone() ? 'today' : 'dayGridMonth,dayGridWeek',
    },
    height: 'auto',
    dayMaxEvents: 2,          // show at most 2 events; remainder becomes "+N more"
    moreLinkClick: 'popover', // replaced in Step 6 with a custom modal handler
    events: () => this.calendarEvents(),
    eventClick: (arg) => this.onEventClick(arg),
    eventClassNames: 'fc-task-event',
}));
Week view: Adding dayGridWeek to the right toolbar button group is all that is needed for the week view. FullCalendar handles it automatically. The plan is designed so this is already wired in from day one.

Step 3 — Build the component template

The template is minimal — it wraps the FullCalendar component and the overflow modal (Step 6). The [options] binding is passed the computed signal value.

<div class="calendar-container"

<!-- Overflow modal (Step 6) -->
@if (overflowModalOpen()) {
    <app-calendar-overflow-modal
        [tasks]="overflowTasks()"
        (taskSelected)="onTaskSelected($event)"
        (close)="overflowModalOpen.set(false)">
    </app-calendar-overflow-modal>

Event click handler

onEventClick(arg: EventClickArg): void {
    const task = arg.event.extendedProps['task'] as Task;
    this.taskSelected.emit(task);
}

Step 6 — Overflow modal: CalendarOverflowModalComponent NEW FILE

When a user clicks "+N others" in a calendar cell, instead of FullCalendar's default popover we open an app-native modal that lists all tasks for that date in a scrollable container. Clicking a task in the modal emits taskSelected and closes the modal.

Wiring the custom more-link click

Override FullCalendar's moreLinkClick with a function callback in CalendarOptions: 

moreLinkClick: (arg: MoreLinkClickArg) => {
    this.overflowDate.set(arg.date);
    this.overflowTasks.set(arg.allSegs.map(s => s.event.extendedProps['task'] as Task));
    this.overflowModalOpen.set(true);
    return 'stop'; // prevent FullCalendar default popover
},

Signals in CalendarViewComponent

overflowModalOpen = signal<boolean>(false);
overflowDate     = signal<Date | null>(null);
overflowTasks    = signal<Task[]>([]);

New component: calendar-overflow-modal/

Files to create:

  • calendar-overflow-modal.component.ts
  • calendar-overflow-modal.component.html
  • calendar-overflow-modal.component.scss

The component accepts tasks = input.required<Task[]>() and date = input<Date | null>(null), emits taskSelected = output<Task>() and close = output<void>(). It renders a centered modal overlay with:

  • A header showing the formatted date (e.g. "May 10") and a close button (Lucide X icon).
  • A scrollable list of task rows — each row shows a checkbox-style completion dot and the task title.
  • Clicking a row emits taskSelected (with the Task) and close.
  • Clicking the backdrop also emits close.
  • Pressing Escape also emits close (keydown listener registered in constructor, removed in ngOnDestroy).

Style the modal using var(--color-bg-secondary) for the card background, var(--radius-lg) for border-radius, and var(--space-4) for padding — consistent with the task-details-panel approach.

The modal body content and its interactions are intentionally minimal at this stage. A future task can enrich it (e.g. inline task completion toggles, add-task shortcut). See comment in the plan footer.

Step 8 — Import FullCalendar global stylesheet EDIT

FullCalendar's base CSS must be imported globally (not just component-scoped) for the plugin styles to load correctly. Add the following import to the top of frontend/src/styles.scss: 

@import '@fullcalendar/core/main.css';
@import '@fullcalendar/daygrid/main.css';
With FullCalendar v6, the package exposes CSS through the same entry points as the JS. If angular.json is configured with "stylePreprocessorOptions" that strips vendor prefixes, verify the import resolves correctly after the first build. An alternative is to add the paths to the "styles" array in angular.json instead.

Step 9 — Verify and polish

After the above steps, perform the following manual checks before considering the feature complete:

  • Month view renders correctly — navigate forward/back through months; tasks appear on their due-date cells; completed tasks appear muted/strikethrough.
  • Week view toggling — clicking the Week button in the toolbar switches to dayGridWeek and back to dayGridMonth correctly.
  • Event click → task-details panel — clicking a task event in any cell opens the TaskDetailsPanelComponent slide-in panel with the correct task loaded.
  • Overflow link → modal — when a cell has more than dayMaxEvents tasks, the "+N others" link opens the overflow modal listing all tasks for that date.
  • Responsive behaviour:
    • Phone (<768 px): single-column header, dayMaxEvents: 1, small font sizes.
    • Tablet (768–1023 px): full toolbar with view switcher, dayMaxEvents: 2.
    • Laptop/Desktop (≥1024 px): toolbar with view switcher and today button, dayMaxEvents: 2, fixed aspect ratio.
  • Search filtering — typing in the global search bar updates the calendar to show only matching tasks in real time (via the reactive calendarEvents computed signal).
  • Accessibility — verify FullCalendar's built-in keyboard navigation works (arrow keys move between cells, Enter selects). The overflow modal traps focus within the dialog and closes on Escape.
  • Global token alignment — compare the rendered calendar against the mockup (full-calendar-with-tasks-mockup.png). Adjust the ::ng-deep overrides from Step 4 until colors, typography, and spacing match.

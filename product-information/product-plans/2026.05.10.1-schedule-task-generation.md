# Schedule Task Generation at User-Timezone Midnight

## Overview

Task list templates already store a `next_generate` timestamp that represents midnight in the user's
local timezone (computed by `RruleService.computeNextGenerate` and stored as a UTC-offset
`OffsetDateTime`). The full generation pipeline (`generateDueTemplates` → `generateFromTemplate`)
is already implemented and handles multi-instance safety via `FOR UPDATE SKIP LOCKED`.

**Two things are missing:**

1. A `TemplateScheduler` class that triggers generation on a recurring schedule.
2. A timezone bug in `TemplateGeneratorService` where `today` (used for due-date offsets) is
   derived from UTC instead of the template's own timezone.

---

## How Timezone-Aware Midnight Generation Works

When a template is created or updated, `RruleService.computeNextGenerate` advances the rrule to
the next occurrence in the user's timezone and converts that midnight to an `OffsetDateTime`:

```java
// e.g. America/New_York (UTC-4 in summer) → midnight ET = 04:00 UTC
next.toLocalDate().atStartOfDay(next.getZone()).toOffsetDateTime();
```

PostgreSQL stores this as a `TIMESTAMPTZ`. `findDueTemplates(now)` then simply checks
`next_generate <= now` (UTC comparison). When the wall clock reaches 04:00 UTC, that template
becomes due — which is exactly midnight for the New York user. No per-timezone cron jobs are
needed; one scheduler checking every minute is sufficient.

## Task 1 — Fix the Timezone Bug in `TemplateGeneratorService`

**File:** `backend/src/main/java/com/killeen/taskflow/components/tasklisttemplate/service/TemplateGeneratorService.java`

`today` is currently computed from the UTC instant, which is wrong for any user whose local
midnight does not coincide with UTC midnight (i.e. everyone not in UTC). At midnight for a user
in UTC-5, `now.toLocalDate()` still returns the previous UTC date.

**Before:**
```java
OffsetDateTime now   = OffsetDateTime.now(ZoneOffset.UTC);
LocalDate      today = now.toLocalDate();
```

**After:**
```java
OffsetDateTime now   = OffsetDateTime.now(ZoneOffset.UTC);
LocalDate      today = now.atZoneSameInstant(ZoneId.of(template.getTimezone())).toLocalDate();
```

Add `java.time.ZoneId` to the existing imports (already in scope from the package). No other
changes to the method are required.

## Task 2 — Create `TemplateScheduler`

**New file:** `backend/src/main/java/com/killeen/taskflow/components/tasklisttemplate/TemplateScheduler.java`

The scheduler runs every minute. Because each template's `next_generate` encodes its own timezone
midnight in UTC, this single cron handles all timezones with at most ~1 minute of delay.

```java
package com.killeen.taskflow.components.tasklisttemplate;

import org.springframework.scheduling.annotation.Scheduled;
import org.springframework.stereotype.Component;

import com.killeen.taskflow.components.tasklisttemplate.service.TaskListTemplateService;

import lombok.RequiredArgsConstructor;
import lombok.extern.slf4j.Slf4j;

@Component
@Slf4j
@RequiredArgsConstructor
public class TemplateScheduler {

    private final TaskListTemplateService taskListTemplateService;

    @Scheduled(cron = "0 * * * * ?")
    public void generateDueTemplates() {
        log.debug("Checking for due templates");
        taskListTemplateService.generateDueTemplates();
    }
}
```

## Existing Safeguards (No Changes Needed)

| Mechanism | Where | What it protects |
|---|---|---|
| `FOR UPDATE SKIP LOCKED` | `TaskListTemplateDbCustomMapper.xml` | Prevents two EC2 instances from processing the same template concurrently |
| `next_generate` comparison in claim | `claimIfStillDue` | Prevents re-processing after `updateSchedule` has advanced `next_generate` |
| `@Transactional` on `generateFromTemplate` | `TemplateGeneratorService` | Holds the DB lock for the entire unit of work |
| Per-template error isolation | `generateDueTemplates` try/catch | One failing template does not abort the rest of the batch |

# Refresh Token: Selector/Validator Refactor

## Why Bother

The current design stores only `sha256(rawToken)` in the DB. The only way to find or delete a row is to present the raw token. If the client loses it, the row is unreachable тАФ it will sit there until `expires_at` passes (currently 7 days).

The selector/validator pattern splits the raw token into two parts: `selector.validator`.

| | Current | Selector/Validator |
|---|---|---|
| Client sends | `rawToken` | `selector.validator` |
| DB stores | `sha256(rawToken)` | `selector` (plain) + `sha256(validator)` |
| Lookup by | full hash (requires raw token) | `selector` (no secret needed) |
| Revoke one session without raw token | тЭМ | тЬЕ via `DELETE WHERE selector = ? AND user_id = ?` |
| Revoke all sessions for a user | partial (`deleteByUserId`) | тЬЕ same |
| Session listing | тЭМ (no safe display key) | тЬЕ expose selector as session ID |
| Cleanup orphans | requires TTL expiry or `deleteByUserId` | тЬЕ by selector тАФ server controls it immediately |

Key point: because the server can address a row by its `selector` alone, you do not need to rely on the client to present the raw token in order to perform any session-management operation.

---

## Bug to Fix During This Refactor

`application.yaml` declares `refresh.expiration: 604800000` with a comment of "7 days in milliseconds", but `RefreshTokenService.createRefreshToken` calls `.plusSeconds(expiration)`. This means tokens expire in ~19 years, not 7 days. Fix the unit during this work: either change the yaml value to `604800` (seconds) and drop the comment, or change the Java call to `.plusMillis(expiration)`.

### Step 4 тАФ Service Refactoring (`RefreshTokenService`)

**Token generation**

Replace `UUID.randomUUID()` with `SecureRandom`-based generation:

```java
private static final int SELECTOR_BYTES  = 16;  // 32 hex chars
private static final int VALIDATOR_BYTES = 32;  // 64 hex chars

public String createRefreshToken(Long userId) {
    SecureRandom rng = new SecureRandom();

    byte[] selectorBytes = new byte[SELECTOR_BYTES];
    rng.nextBytes(selectorBytes);
    String selector = HexFormat.of().formatHex(selectorBytes);

    byte[] validatorBytes = new byte[VALIDATOR_BYTES];
    rng.nextBytes(validatorBytes);
    String validatorRaw = HexFormat.of().formatHex(validatorBytes);
    String validatorHashed = hashUtils.sha256(validatorRaw);

    OffsetDateTime expirationTime = OffsetDateTime.now(ZoneOffset.UTC).plusSeconds(expiration);

    RefreshToken token = RefreshToken.builder()
        .userId(userId)
        .selector(selector)
        .token(validatorHashed)
        .expiresAt(expirationTime)
        .build();

    refreshTokenRepository.save(token);

    return selector + "." + validatorRaw;  // this is what gets sent to the client
}
```

**Token validation** (`findByToken`)

Rename to `validateToken(String rawToken)` or keep the name but update the logic:

```java
public RefreshToken findByToken(String rawToken) {
    String[] parts = rawToken.split("\\.", 2);
    if (parts.length != 2) {
        throw new InvalidRefreshTokenException(env.getProperty("refreshtoken.token.null.or.blank"));
    }

    String selector = parts[0];
    String validatorRaw = parts[1];

    RefreshToken stored = refreshTokenRepository.findBySelector(selector)
        .orElseThrow(() -> new RefreshTokenNotFoundException(
            env.getProperty("refreshtoken.token.not.found")));

    String expectedHash = hashUtils.sha256(validatorRaw);
    if (!MessageDigest.isEqual(
            expectedHash.getBytes(StandardCharsets.UTF_8),
            stored.getToken().getBytes(StandardCharsets.UTF_8))) {
        throw new InvalidRefreshTokenException(env.getProperty("refreshtoken.token.invalid.or.unknown"));
    }

    return stored;
}
```

The constant-time compare (`MessageDigest.isEqual`) prevents timing attacks where an attacker probes for a valid selector by measuring response time.

**Token deletion** (`deleteByToken`)

```java
public void deleteByToken(String rawToken) {
    String selector = rawToken.split("\\.", 2)[0];
    refreshTokenRepository.deleteBySelector(selector);
}
```

**New method: deleteByUserId**

```java
public void deleteAllForUser(Long userId) {
    refreshTokenRepository.deleteByUserId(userId);
}
```

### Step 5 тАФ Repository Changes (`RefreshTokenRepository`)

Add three methods:

```java
public Optional<RefreshToken> findBySelector(String selector) {
    RefreshTokenDbExample example = new RefreshTokenDbExample();
    example.createCriteria().andSelectorEqualTo(selector);
    return mapper.selectByExample(example).stream()
        .findFirst()
        .map(converter::toDto);
}

public void deleteBySelector(String selector) {
    RefreshTokenDbExample example = new RefreshTokenDbExample();
    example.createCriteria().andSelectorEqualTo(selector);
    mapper.deleteByExample(example);
}

public void deleteByUserId(Long userId) {
    RefreshTokenDbExample example = new RefreshTokenDbExample();
    example.createCriteria().andUserIdEqualTo(userId);
    mapper.deleteByExample(example);
}

public void deleteExpired() {
    RefreshTokenDbExample example = new RefreshTokenDbExample();
    example.createCriteria().andExpiresAtLessThan(OffsetDateTime.now(ZoneOffset.UTC));
    mapper.deleteByExample(example);
}
```

Remove the now-redundant `findByToken(String hashedToken)` if nothing else calls it.

### Step 6 тАФ Controller Endpoints

Wire `RefreshTokenService` into `AuthController` (or a new `SessionController` if you want separation). Minimum endpoints needed to make refresh tokens useful:

| Method | Path | Description |
|---|---|---|
| `POST` | `/auth/refresh` | Accept `selector.validator` тЖТ validate тЖТ issue new JWT + rotate refresh token |
| `POST` | `/auth/logout` | Accept `selector.validator` тЖТ delete the token row |
| `DELETE` | `/auth/sessions` | Authenticated тАФ delete all sessions for the current user |

Future (not blocking):

| Method | Path | Description |
|---|---|---|
| `GET` | `/auth/sessions` | Return list of `{ selector, createdAt, expiresAt }` for current user |
| `DELETE` | `/auth/sessions/{selector}` | Revoke one specific session by selector |

For `POST /auth/refresh`, the recommended pattern is **token rotation**: on a successful refresh, delete the old token and issue a brand new `selector.validator` pair. Return the new refresh token to the client alongside the new JWT.

### Step 7 тАФ Cleanup Job

Add a `@Scheduled` component to periodically purge expired rows.

```java
@Component
@RequiredArgsConstructor
public class RefreshTokenCleanupJob {

    private final RefreshTokenRepository refreshTokenRepository;

    @Scheduled(cron = "${refresh.cleanup.cron:0 0 * * * *}")  // default: top of every hour
    public void deleteExpiredTokens() {
        refreshTokenRepository.deleteExpired();
    }
}
```

Add to `application.yaml`:

```yaml
refresh:
  expiration: 604800   # 7 days in SECONDS (fix from ms)
  cleanup.cron: "0 0 * * * *"
```

Enable scheduling in your Spring Boot app (if not already present):

```java
@EnableScheduling
```

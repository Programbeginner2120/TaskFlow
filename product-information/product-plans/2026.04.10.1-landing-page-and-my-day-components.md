## Implementation Standards

### Accessibility

All shared components (`SearchBarComponent`, `TaskDetailsPanelComponent`) must be fully accessible:

- Semantic HTML elements used where appropriate (`<button>`, `<input>`, `<textarea>`, `<label>`)
- All interactive elements have descriptive `aria-label` or `aria-labelledby` attributes
- The task details panel uses `role="dialog"` with `aria-modal="true"` and `aria-labelledby` pointing to its title
- Focus is trapped inside the task details panel while it is open; focus returns to the triggering element on close
- The panel closes on `Escape` key press
- All color combinations meet WCAG 2.1 AA contrast ratios (4.5:1 for text, 3:1 for UI elements), consistent with PRD req DS-1
- Checkbox and toggle controls have visible focus indicators

### Skills & Best Practices

When implementing each step, load and follow the relevant VS Code Copilot skills for Angular and frontend best practices before writing any code. This ensures implementation quality is consistent with tested patterns for this project.

## On the Sidebar Question

**Short answer: Do NOT reuse the existing `SidebarComponent` for the task details panel.**

The existing `app-sidebar` is a *left-side navigation drawer* — it manages nav items, has horizontal collapse/expand behavior, and uses `--sidebar-width` positioning. The task details panel is a *right-side content overlay* with domain-specific task fields (title, due date, notes, subtasks). Trying to force the nav sidebar into a task details role would mean fighting its collapse logic, positioning, CSS variables, and content projection API at every step. A separate `TaskDetailsPanelComponent` is the right call — it also makes the nav sidebar and the detail panel independently evolvable.

- CSS: `position: fixed; right: 0; top: 0; height: 100vh; width: 420px`; semi-transparent backdrop behind panel; clicking backdrop emits `close`; slide-in animation via `transform: translateX()` CSS transition
- Reuses existing `SelectComponent` (`app-select`) for the list dropdown

## Phase 4 — Layout Shell & My Day View

### Step 7: Redesign LandingPageComponent

Rewrite all three files in `frontend/src/app/pages/landing-page/`

*Depends on Steps 5 and 6*

- New signals:
  - `currentView = signal<'my-day' | 'upcoming' | 'calendar' | 'list'>('my-day')`
  - `selectedListId = signal<number | null>(null)`
  - `searchQuery = signal<string>('')`
  - `selectedTask = signal<Task | null>(null)`
- Injects: `TaskListService` (sidebar list items)
- Sidebar nav items: `navItemRouteFn` calls `currentView.set('my-day')` etc.
- Sidebar list items: generated from `TaskListService.lists()` and projected into `[sidebar-content]` slot
- Template structure:
  ```
  <div class="layout">
    <app-landing-page-header [(searchQuery)]="searchQuery" />
    <div class="layout-body">
      <app-sidebar [navItems]="navItems()" [collapsible]="true">
        <ng-container sidebar-content> ...list items... </ng-container>
      </app-sidebar>
      <main class="main-content">
        @switch (currentView()) {
          @case ('my-day') {
            <app-my-day
              [searchQuery]="searchQuery()"
              (taskSelected)="selectedTask.set($event)" />
          }
          @default { <div>Coming soon</div> }
        }
      </main>
    </div>
    <app-task-details-panel
      [task]="selectedTask()"
      (close)="selectedTask.set(null)" />
  </div>
  ```
- CSS: full-viewport layout; header fixed at top; body is a flex row (sidebar + main); main scrolls independently

### Step 8: MyDayComponent

Create `frontend/src/app/pages/my-day/` (3 files: `.ts`, `.html`, `.scss`)

*Can be built in parallel with Step 7's template work*

- Selector: `app-my-day`
- Input: `searchQuery = input<string>('')`
- Output: `taskSelected = output<Task>()`
- Injects: `TaskService`, `TaskListService`
- Computed signals:
  - `todaysTasks()` — `TaskService.tasks()` filtered where `dueDate` matches today by calendar day
  - `filteredTasks()` — `todaysTasks()` filtered by case-insensitive substring match on `searchQuery()`
  - `sortedTasks()` — `filteredTasks()` sorted incomplete first, completed last
  - `incompleteCount()` — count of non-completed tasks in `filteredTasks()`
- Local signal: `newTaskTitle = signal<string>('')`
- Methods: `addTask()` (calls `taskService.addTask()`, clears input), `toggleTask(id)`, `openDetails(task)` (emits `taskSelected`)
- Template (matches mockup):
  - `<h1>My Day</h1>` + `<p>{{ incompleteCount() }} tasks remaining</p>`
  - Add task bar: `<input>` with `+ Add a task...` placeholder + calendar icon button (date picker future scope)
  - `@for (task of sortedTasks(); track task.id)` — task row:
    - Completion circle button: hollow circle when incomplete, filled indigo check when complete — calls `toggleTask(task.id)`
    - Task title (strikethrough + reduced opacity when `task.completed`)
    - Metadata line: "Today" · colored dot + list name · subtask progress badge (e.g. `1/3`) if subtasks exist
    - Chevron right icon — calls `openDetails(task)`

## Acceptance Criteria

1. `/landing-page` loads showing sidebar + header + My Day content
2. Only today-dated mock tasks appear (Review Q2 roadmap presentation, Fix login page bug, Morning jog 5km)
3. Typing in the search bar filters the task list in real time
4. Clicking the completion circle toggles strikethrough + moves the task to the bottom of the list
5. Clicking a task row opens the task details panel from the right
6. Task details panel shows correct data — title, due date (April 10th, 2026), list (Work), notes, subtasks with "SUBTASKS (1/3)" progress
7. "Check timeline accuracy" subtask appears checked/crossed out; other two are unchecked
8. Subtask checkboxes independently toggle completion
9. "+ Add subtask..." input creates a new subtask on Enter
10. X button and backdrop click both close the task details panel
11. Theme toggle in header switches between dark and light mode without layout shift
12. Sidebar collapse button collapses the left navigation

# Datepicker Refactor — Explicit Position Input

## Problem with the Current Approach

The component currently runs `computePosition()` once when the popup opens. It measures viewport space and picks a vertical (`'below' | 'above'`) and horizontal (`'start' | 'end'`) value, storing them in a `popupPosition` signal. This approach has several issues:

- **Hardcoded guesses**: `POPUP_HEIGHT = 300` and `POPUP_WIDTH = 256` are estimates. If the popup's actual rendered size differs, the calculation is wrong.
- **Wrong responsibility**: The component cannot know the caller's layout context (e.g., inside a scrollable container, near a sticky header, etc.). The caller always knows better.
- **Dead complexity**: `triggerRef`, `computePosition()`, two private constants, and the internal signal are all infrastructure serving a calculation that should not exist inside this component.
- **Not composable**: Every callsite gets the same auto-detection logic with no way to override it.

## Proposed Solution

Accept a `position` input typed to a `DatepickerPosition` string-union. The component applies a single CSS modifier class derived directly from that input. No measurements, no signals, no constants.

### `DatepickerPosition` Type

Define a string-union type (not an enum, to stay idiomatic with Angular `input()` and template literals):

```ts
export type DatepickerPosition =
  | 'bottom-right'   // default — below the trigger, right-aligned
  | 'bottom-left'    // below the trigger, left-aligned
  | 'top-right'      // above the trigger, right-aligned
  | 'top-left'       // above the trigger, left-aligned
  | 'right'          // to the right of the trigger, top-aligned
  | 'left'           // to the left of the trigger, top-aligned
  | 'bottom'         // below the trigger, centered
  | 'top';           // above the trigger, centered
```

Place this in the existing `datepicker.interface.ts` file alongside `CalendarDay`.

---

## Implementation Steps

### 1. Add `DatepickerPosition` to the interface file

Add the type export to `src/app/shared/interfaces/datepicker.interface.ts`.

### 2. Update `datepicker.component.ts`

**Remove:**
- `private readonly triggerRef = viewChild<ElementRef<HTMLButtonElement>>('trigger');`
- `popupPosition = signal<...>(...);`
- `private readonly POPUP_HEIGHT = 300;`
- `private readonly POPUP_WIDTH = 256;`
- `private computePosition(): void { ... }`
- The `computePosition()` call inside `toggle()`
- `viewChild` from the imports list

**Add:**
- `position = input<DatepickerPosition>('bottom-right');`
- Import `DatepickerPosition` from the interface file

### 3. Update `datepicker.component.html`

Replace the two dynamic class bindings:

```html
<!-- before -->
[class.datepicker-popup--above]="popupPosition().vertical === 'above'"
[class.datepicker-popup--start]="popupPosition().horizontal === 'start'"

<!-- after -->
[class]="'datepicker-popup datepicker-popup--' + position()"
```

Remove the `#trigger` template reference variable from the trigger button (no longer needed).

### 4. Update `datepicker.component.scss`

Remove the two existing modifier classes (`&--above`, `&--start`) and replace with one rule per position value. The base `.datepicker-popup` rule keeps `position: absolute` and `z-index` but removes the default offset/anchor values (those move into each modifier):

```scss
// Base — layout only, no positional offsets
.datepicker-popup {
    position: absolute;
    z-index: 200;
    width: 256px;
    // ... visual styles unchanged ...
}

// ── Vertical: below ──────────────────────────────────────────
.datepicker-popup--bottom-right {
    top: calc(100% + var(--space-2));
    right: 0;
}
.datepicker-popup--bottom-left {
    top: calc(100% + var(--space-2));
    left: 0;
}
.datepicker-popup--bottom {
    top: calc(100% + var(--space-2));
    left: 50%;
    transform: translateX(-50%);
}

// ── Vertical: above ──────────────────────────────────────────
.datepicker-popup--top-right {
    bottom: calc(100% + var(--space-2));
    right: 0;
}
.datepicker-popup--top-left {
    bottom: calc(100% + var(--space-2));
    left: 0;
}
.datepicker-popup--top {
    bottom: calc(100% + var(--space-2));
    left: 50%;
    transform: translateX(-50%);
}

// ── Horizontal: to the right ─────────────────────────────────
.datepicker-popup--right {
    left: calc(100% + var(--space-2));
    top: 0;
}

// ── Horizontal: to the left ──────────────────────────────────
.datepicker-popup--left {
    right: calc(100% + var(--space-2));
    top: 0;
}
```

### 5. Update all callsites

Search for `<app-datepicker` across the frontend and add an explicit `position` binding where the default `'bottom-right'` is not appropriate. Since the existing auto-detection defaulted to `'below'` + `'end'` (right), `'bottom-right'` is a behaviorally equivalent default for all current usages — no callsites need changes unless a specific layout requires a different position.

---

## What is NOT Changing

- All date logic (`calendarDays`, `minDate`, `maxDate`, `value`, navigation, disabled state)
- The trigger button's visual design
- The popup's internal layout and styles
- Keyboard and click-outside behaviour
- The `CalendarDay` interface

# Plan: Rich Seed Data with Realistic Timestamps

**Date:** 2026-05-24

---

## Problem

The current `DataSeeder` calls `taskListService.createTaskList()` and `taskService.createTask()`,
both of which stamp `createdAt = OffsetDateTime.now()`. Every seed record therefore lands at the
same instant — the moment the app started. Seed data useful for UI testing needs:

- Many tasks spread across weeks or months.
- A believable mix of `createdAt` dates.
- Some tasks completed, with `completedAt` dates that fall after `createdAt`.
- Variety in list, due-date, and note presence.

---

## Options

### Option A — JdbcTemplate Timestamp Patching ✅ Recommended

**How it works**

Keep the service-layer approach for all inserts (BCrypt and field encryption remain correct).
After every create call, record the returned `id`. At the end of seeding, run a batch of
`UPDATE` statements via an injected `JdbcTemplate` to overwrite `created_at`, `updated_at`,
and `completed_at` with the historical dates we actually want.

```java
jdbcTemplate.update(
    "UPDATE tasks SET created_at = ?, updated_at = ?, completed_at = ? WHERE id = ?",
    createdAt, updatedAt, completedAt, taskId
);
```

For task lists the same applies to `created_at` / `updated_at`.

**Pros**
- Zero changes to any service, repository, or model.
- Encryption and password hashing handled correctly by the service layer.
- `completedAt` can be set directly — no need to call `updateTask()` first.
- `JdbcTemplate` is already on the classpath (Spring JDBC, included with Spring Data).

**Cons**
- Two-pass approach: create then patch. Slightly more code in `DataSeeder`.
- If a future migration renames a column, the hardcoded SQL column name breaks (low risk
  for a local-only seeder).

---

### Option B — Direct Repository Saves with `TextEncryptor` Injection

**How it works**

Skip the service layer entirely. Inject `TaskRepository`, `TaskListRepository`,
`TextEncryptor` (or `TaskEncryptionHelper` / `TaskListEncryptionHelper`), and
`PasswordEncoder`. Build full model objects with arbitrary timestamps, manually
encrypt the fields that require it, and call `repository.save()` directly.

**Pros**
- Single-pass; total control over every field including timestamps.

**Cons**
- Must understand and replicate every encryption call the service makes (which fields
  are encrypted, in which order, via which helper). If the helper does more than a
  plain `encrypt()` call (e.g. null guards, trimming), the seed data will be wrong.
- Any future change to the encryption pipeline silently breaks the seeder.
- Couples `DataSeeder` to implementation details of the encryption helpers.

---

### Option C — Injectable `Clock` Bean (Local Profile Override)

**How it works**

Refactor every service method that calls `OffsetDateTime.now()` to instead call
`clock.instant()` on an injected `java.time.Clock`. In the `local` Spring profile,
expose a mutable `Clock` bean. `DataSeeder` advances the clock to the desired date
before each service call, then resets it.

**Pros**
- Architecturally cleanest — services remain unaware of seed concerns.
- Makes services properly testable with a fixed clock.

**Cons**
- Requires touching every service method that sets timestamps — significant refactor
  for what is purely a dev-convenience feature.
- Shared mutable clock state is fragile if any service caches the clock or reads it
  on a background thread.
- High effort; low payoff compared to Option A.

---

### Option D — Python psycopg2 Scripts (Plan A Revisited)

**How it works**

Use the existing `scripts/python/` infrastructure for direct DB inserts. Timestamps
are plain SQL values, so full historical spread is trivial. Python's `bcrypt` library
handles password hashing. Spring field encryption (PBKDF2-SHA1 key derivation + AES-CBC)
would need to be replicated with the `cryptography` library.

**Pros**
- Total freedom: any number of records, any date range, easy to re-run or modify.
- No Spring context needed; scripts are standalone.

**Cons**
- Spring's `Encryptors.text()` uses PBKDF2-SHA1 (1024 iterations, 256-bit key) then
  AES/CBC with a random 8-byte IV, hex-encoded. Replicating this correctly in Python
  requires careful parameter matching; a single mismatch produces silently corrupt data
  that the app will fail to decrypt at read time.
- The encryption key and salt live in `.env` — Python scripts must load them and the
  implementation is tied to the Spring version of that algorithm.
- This was previously considered and rejected for the same reason.

---

### Option E — Liquibase "local" Context Changeset

**How it works**

Add a Liquibase changeset tagged with `context: local`. Pre-compute a BCrypt hash
offline and hardcode it. Pre-compute ciphertexts for all encrypted fields using the
**standardised** local key/salt from `docker-compose.local.yml`.

**Pros**
- Pure SQL; trivially sets any timestamps.

**Cons**
- Pre-computed ciphertexts are correct only as long as the local encryption key/salt
  stay the same. Any `.env` change (per-developer customization, key rotation) corrupts
  every seed row silently.
- Hardcoded ciphertext strings are opaque — impossible to audit or update.
- BCrypt hash commits a known credential to version control.

---

## Recommendation: Option A — JdbcTemplate Timestamp Patching

No service refactoring. No encryption reimplementation. `JdbcTemplate` is already
available as a Spring bean. `DataSeeder` does all the work itself, scoped to
`@Profile("local")`.

---

## Implementation Plan

### Step 1 — Update `DataSeeder` dependencies

Inject `JdbcTemplate` alongside the existing beans.

```java
private final JdbcTemplate jdbcTemplate;
```

### Step 2 — Define seed timeline constants

```java
private static final OffsetDateTime T_BASE =
    OffsetDateTime.of(2026, 3, 1, 9, 0, 0, 0, ZoneOffset.UTC);
```

All historical dates are offsets from `T_BASE`.

### Step 3 — Return IDs from helper methods

Change `createList()` and `createTask()` to return the created entity so the caller
has access to the `id`. The current service methods already return the full object.

### Step 4 — Patch timestamps after creation

After all service calls, run a batched timestamp patch:

```java
private void patchTimestamps(Long taskId, OffsetDateTime createdAt,
                             OffsetDateTime completedAt) {
    OffsetDateTime updatedAt = completedAt != null ? completedAt : createdAt;
    jdbcTemplate.update(
        "UPDATE tasks SET created_at = ?, updated_at = ?, completed_at = ? WHERE id = ?",
        createdAt, updatedAt, completedAt, taskId
    );
}

private void patchListTimestamp(Long listId, OffsetDateTime createdAt) {
    jdbcTemplate.update(
        "UPDATE task_lists SET created_at = ?, updated_at = ? WHERE id = ?",
        createdAt, createdAt, listId
    );
}
```

### Step 5 — Rewrite `DataSeeder` seed data

Replace the flat task list with a structured seed table (see below), then build
`createSeedTasks()` to iterate over it.

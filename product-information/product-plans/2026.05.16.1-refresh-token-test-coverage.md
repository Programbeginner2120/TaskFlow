### `createRefreshToken`

| Test | Scenario | Assertions |
|---|---|---|
| `createRefreshToken_returnsRawTokenInSelectorDotValidatorFormat` | Happy path | Returned string contains exactly one `.`; parts before and after the dot are non-blank hex strings; part before dot is 32 hex chars (16 bytes), part after is 64 hex chars (32 bytes) |
| `createRefreshToken_savesHashedValidatorNotRawValidator` | Security property | Capture the `RefreshToken` passed to `refreshTokenRepository.save()` via `ArgumentCaptor`. Assert that `token` field (the stored hash) is NOT equal to the raw validator but IS equal to `sha256(rawValidator)` |
| `createRefreshToken_setsCorrectUserId` | Persistence | Captured `RefreshToken.userId` equals the userId argument |
| `createRefreshToken_setsExpiresAtInFuture` | TTL | Captured `RefreshToken.expiresAt` is after `OffsetDateTime.now(UTC)` |
| `createRefreshToken_eachCallProducesUniqueTokens` | Uniqueness | Call twice with the same userId; assert both returned strings are different |

## Implementation Notes

- Use `ArgumentCaptor<RefreshToken>` in service tests that need to inspect what was saved to the repository.
- The `HashUtils` component has no external dependencies beyond the JVM — instantiate it directly in service tests rather than mocking it, so the hash-verification logic is exercised end-to-end within the service test (avoid mocking the security implementation).
- For controller tests, inject `RefreshTokenService` as `@MockitoBean` and fully mock its behavior — controller tests are about HTTP contract, not token logic.
- Follow the established naming convention: `methodName_scenario_expectedOutcome`.
- Use `@DisplayName` annotations for any test whose name alone doesn't fully express the security intent (e.g., the hash-not-stored-as-plaintext test).

# 2026.05.16.2 ŌĆö Integrate Refresh Token with Frontend

## Overview

The backend refresh token infrastructure is fully built (`RefreshTokenService`, `RefreshTokenRepository`, `RefreshTokenCleanupJob`, `POST /auth/refresh`, `POST /auth/logout`). However, there are **two backend gaps** and **three frontend gaps** that must be closed before end-to-end token refresh works.

---

## Current State

### Backend ŌĆö What Exists

| Endpoint | Purpose |
|---|---|
| `POST /auth/login` | Returns `{ token, expiresIn }` ŌĆö **no refresh token** |
| `POST /auth/refresh` | Accepts `{ refreshToken }`, rotates and returns `{ token, expiresIn, refreshToken }` |
| `POST /auth/logout` | Accepts `{ refreshToken }`, deletes it from DB |
| `DELETE /auth/sessions` | Deletes all refresh tokens for the authenticated user |

### Frontend ŌĆö What Exists

- `AuthService` stores the JWT in `localStorage` under `'auth_token'`. No concept of refresh tokens.
- `AuthInterceptor` attaches the JWT and **immediately calls `authService.logout()` on any 401** ŌĆö no retry.
- `authGuard` synchronously checks `isAuthenticated()` (which inspects JWT expiry). No refresh attempt.
- `auth.interface.ts` ŌĆö `LoginResponse` only models `{ token, expiresIn }`.

---

## Gaps and Required Changes

### Gap 1 ŌĆö Backend: `POST /auth/login` Does Not Return a Refresh Token

**Location:** `UserService.authenticate()` + `LoginResponse.java`

`UserService.authenticate()` builds a `LoginResponse` with only `token` and `expiresIn`. It never calls `refreshTokenService.createRefreshToken()`. The frontend therefore never receives a refresh token after login, making the entire refresh flow impossible to start.

**Fix:**
1. Add `private String refreshToken;` to `LoginResponse.java`.
2. In `UserService.authenticate()`, inject `RefreshTokenService`, call `createRefreshToken(user.getId())` after generating the JWT, and include the raw token in the `LoginResponse`.

---

### Gap 2 ŌĆö Frontend: `LoginResponse` Interface Missing `refreshToken`

**Location:** `frontend/src/app/interfaces/auth.interface.ts`

The TypeScript `LoginResponse` interface mirrors the old backend shape. It must be updated to include the refresh token, and new interfaces for the refresh flow are needed.

**Fix ŌĆö add to `auth.interface.ts`:**
```typescript
export interface LoginResponse {
    token: string;
    expiresIn: number;
    refreshToken: string; // add this
}

export interface RefreshRequest {
    refreshToken: string;
}

export interface RefreshResponse {
    token: string;
    expiresIn: number;
    refreshToken: string;
}
```

### Gap 3 ŌĆö Frontend: `AuthService` Has No Refresh Token Lifecycle

**Location:** `frontend/src/app/services/auth.service.ts`

The service has no awareness of refresh tokens. `login()` only stores the JWT. `logout()` only clears local state ŌĆö it never calls the backend to invalidate the DB record. There is no `refresh()` method.

**Fix ŌĆö three changes inside `AuthService`:**

**3a. Store the refresh token after login:**
```typescript
private readonly REFRESH_TOKEN_KEY = 'refresh_token';
private readonly _refreshToken = signal<string | null>(
    localStorage.getItem(this.REFRESH_TOKEN_KEY)
);

// In login():
tap(response => {
    this.setToken(response.token);
    this.setRefreshToken(response.refreshToken);
})

private setRefreshToken(token: string): void {
    localStorage.setItem(this.REFRESH_TOKEN_KEY, token);
    this._refreshToken.set(token);
}

private clearRefreshToken(): void {
    localStorage.removeItem(this.REFRESH_TOKEN_KEY);
    this._refreshToken.set(null);
}

getRefreshToken(): string | null {
    return this._refreshToken();
}
```

**3b. Invalidate the refresh token on logout:**
```typescript
logout(): void {
    const refreshToken = this.getRefreshToken();
    if (refreshToken) {
        // Fire-and-forget ŌĆö best effort server-side invalidation
        this.http.post(`${this.API_URL}/logout`, { refreshToken }).subscribe();
    }
    this.logoutSubject.next();
    this.clearToken();
    this.clearRefreshToken();
    this._currentUser.set(null);
    this.router.navigate(['/login']);
}
```

**3c. Add a `refresh()` method:**
```typescript
refresh(): Observable<RefreshResponse> {
    const refreshToken = this.getRefreshToken();
    if (!refreshToken) {
        return throwError(() => new Error('No refresh token available'));
    }
    return this.http.post<RefreshResponse>(`${this.API_URL}/refresh`, { refreshToken }).pipe(
        tap(response => {
            this.setToken(response.token);
            this.setRefreshToken(response.refreshToken);
        })
    );
}
```

> **Security note:** `localStorage` is used here to match the existing JWT storage pattern. If the threat model changes in the future, both tokens should migrate to `HttpOnly` cookies to eliminate XSS exposure.

### Gap 4 ŌĆö Frontend: `AuthInterceptor` Logs Out Immediately on 401

**Location:** `frontend/src/app/interceptors/auth-interceptor.ts`

This is the most complex change. The interceptor currently calls `authService.logout()` the moment any 401 arrives. With refresh tokens, a 401 should trigger a token refresh attempt. The key complication is **concurrent requests** ŌĆö if three requests all return 401 at the same time, only one refresh call should be made; the other two should queue and retry once the refresh succeeds.

**The pattern uses two signals on `AuthService` to coordinate the lock, bridged to Observables inside the interceptor via `toObservable()` from `@angular/core/rxjs-interop`:**

```typescript
// In AuthService ŌĆö signals for refresh-lock coordination
private readonly _isRefreshing = signal(false);
private readonly _refreshedToken = signal<string | null>(null);

readonly isRefreshing = this._isRefreshing.asReadonly();
readonly refreshedToken = this._refreshedToken.asReadonly();

// Called by the interceptor to manage lock state
startRefresh(): void {
    this._isRefreshing.set(true);
    this._refreshedToken.set(null);
}

completeRefresh(token: string): void {
    this._isRefreshing.set(false);
    this._refreshedToken.set(token);
}

failRefresh(): void {
    this._isRefreshing.set(false);
}
```

`startRefresh()`, `completeRefresh()`, and `failRefresh()` keep mutation off the interceptor and inside the service where state belongs. The interceptor reads the signals and converts them to Observables only where the RxJS pipeline requires it.

**Updated `AuthInterceptor`:**
```typescript
import { toObservable } from '@angular/core/rxjs-interop';

export const AuthInterceptor: HttpInterceptorFn = (req, next) => {
    const authService = inject(AuthService);
    const token = authService.getToken();

    // Skip adding auth header for the refresh endpoint itself
    const authReq = token && !req.url.includes('/auth/refresh')
        ? req.clone({ setHeaders: { Authorization: `Bearer ${token}` } })
        : req;

    return next(authReq).pipe(
        catchError((error: HttpErrorResponse) => {
            if (error.status !== 401 || req.url.includes('/auth/refresh')) {
                return throwError(() => error);
            }

            if (authService.isRefreshing()) {
                // Queue: convert the refreshedToken signal to an Observable and wait
                return toObservable(authService.refreshedToken).pipe(
                    filter(t => t !== null),
                    take(1),
                    switchMap(newToken =>
                        next(req.clone({ setHeaders: { Authorization: `Bearer ${newToken}` } }))
                    )
                );
            }

            authService.startRefresh();

            return authService.refresh().pipe(
                switchMap(response => {
                    authService.completeRefresh(response.token);
                    return next(req.clone({ setHeaders: { Authorization: `Bearer ${response.token}` } }));
                }),
                catchError(refreshError => {
                    authService.failRefresh();
                    authService.logout();
                    return throwError(() => refreshError);
                })
            );
        })
    );
};
```

### Gap 5 ŌĆö Frontend: `authGuard` Does Not Proactively Refresh

**Location:** `frontend/src/app/guards/auth.guard.ts`

Currently the guard checks `isAuthenticated()` synchronously. If the JWT is expired but a valid refresh token exists in storage, the guard will redirect the user to `/login` even though they could seamlessly get a new JWT. The interceptor would normally recover mid-flight, but the guard fires before any HTTP call is made.

**Fix ŌĆö make the guard async and attempt a refresh if the JWT is expired:**
```typescript
export const authGuard: CanActivateFn = () => {
    const authService = inject(AuthService);
    const router = inject(Router);

    if (authService.isAuthenticated()) {
        return true;
    }

    // JWT expired ŌĆö try to refresh before rejecting
    if (authService.getRefreshToken()) {
        return authService.refresh().pipe(
            map(() => true),
            catchError(() => {
                authService.logout();
                return of(router.createUrlTree(['/login']));
            })
        );
    }

    return router.createUrlTree(['/login']);
};
```

## Implementation Order

1. **Backend** ŌĆö Add `refreshToken` to `LoginResponse` + update `UserService.authenticate()` (Gap 1). This unblocks all frontend work.
2. **Interfaces** ŌĆö Add `refreshToken` to `LoginResponse` TS interface, add `RefreshRequest` / `RefreshResponse` (Gap 2).
3. **AuthService** ŌĆö Add refresh token storage, update `login()`, update `logout()`, add `refresh()` + refresh-lock state (Gap 3).
4. **AuthInterceptor** ŌĆö Replace immediate-logout-on-401 with refresh-and-retry logic (Gap 4).
5. **authGuard** ŌĆö Make guard async and attempt refresh on expired JWT (Gap 5).

## Token Storage Strategy

Both tokens are currently stored in `localStorage` to be consistent with the existing approach. The risks are:

| Token | Storage | XSS Risk | CSRF Risk |
|---|---|---|---|
| JWT (short-lived) | `localStorage` | Yes | No |
| Refresh token (long-lived) | `localStorage` | Yes | No |

If the application moves to `HttpOnly` cookies in the future, both tokens should be migrated together, and the `withCredentials: true` flag would need to be set on HTTP requests.

## Acceptance Criteria

- [ ] Login response includes a refresh token
- [ ] Refresh token is stored locally after login
- [ ] When a JWT expires, the interceptor silently refreshes and retries the original request
- [ ] Concurrent 401s result in exactly one refresh call; other requests queue and retry
- [ ] Logout sends the refresh token to the backend for server-side invalidation
- [ ] A user navigating to a protected route with an expired JWT but valid refresh token is not redirected to login
- [ ] If refresh fails (expired/revoked), the user is redirected to login and all local tokens are cleared
